## 🧠 AI/ML 前沿论文

**[Maestro: Reinforcement Learning to Orchestrate Hierarchical Model-Skill Ensembles](https://huggingface.co/papers/2605.22177)**
🔬 突破：推翻了“单一LLM+固定路由”的Agent架构假设，提出**用强化学习动态编排多个LLM和技能模块的层级组合**。在跨领域任务（代码生成+数学推理+知识问答）上，Maestro相比GPT-4单模型路由，任务成功率提升约22%，且推理成本降低约35%（因只在必要时调用高成本模型）。
⚙️ 工程影响：这意味着Agent系统的架构将从“选一个最强的LLM”转向“维护一个模型池+RL路由策略”。团队需要开始评估：是训练一个专用路由模型，还是用现成LLM做路由决策？Maestro的RL训练代码开源后，可直接用于微调路由策略。

**[KVServe: Service-Aware KV Cache Compression for Communication-Efficient Disaggregated LLM Serving](https://huggingface.co/papers/2605.13734)**
🔬 突破：推翻了“KV Cache压缩率应固定”的假设，提出**根据当前服务负载、带宽和SLO动态调整压缩率**。在PD分离架构中，相比静态压缩（如固定50%压缩率），KVServe在混合负载下P99延迟降低约40%，且压缩率在30%-70%之间动态波动。
⚙️ 工程影响：对于部署了PD分离推理的团队，这意味着KV Cache压缩不再是“设置一次就不管”的配置项，而需要引入一个实时监控+动态调整的控制平面。本周可评估：在现有推理框架中，KV Cache传输是否已成为瓶颈？如果是，KVServe的“服务感知”思路比单纯增加带宽更经济。

**[π-Bench: Evaluating Proactive Personal Assistant Agents in Long-Horizon Workflows](https://huggingface.co/papers/2605.14678)**
🔬 突破：推翻了“Agent能力=单轮指令跟随”的评估方式，提出**在长程多轮交互中评估Agent的“主动发现隐含需求”能力**。在π-Bench的测试中，当前最强Agent（如OpenClaw）在“用户只说‘帮我安排行程’但未提预算/偏好”的场景下，主动追问率仅约35%，而人类助理约为80%。
⚙️ 工程影响：这意味着当前Agent的“被动执行”模式在真实场景中可能严重不足。团队在构建个人助手Agent时，需要将“意图挖掘”作为独立模块设计，而非依赖用户一次性说清需求。π-Bench的测试集可作为评估Agent“主动性”的基准。

## 🧠 AI/ML Frontier Papers

**[Maestro: Reinforcement Learning to Orchestrate Hierarchical Model-Skill Ensembles](https://huggingface.co/papers/2605.22177)**
🔬 Breakthrough: It overturns the assumption of a "single LLM + fixed routing" Agent architecture, proposing **using reinforcement learning to dynamically orchestrate hierarchical combinations of multiple LLMs and skill modules**. On cross-domain tasks (code generation + mathematical reasoning + knowledge QA), Maestro improves task success rate by about 22% compared to GPT-4 single-model routing, and reduces inference cost by about 35% (by only calling high-cost models when necessary).
⚙️ Engineering Impact: This means the architecture of Agent systems will shift from "choosing one strongest LLM" to "maintaining a model pool + RL routing strategy". Teams need to start evaluating: is it better to train a dedicated routing model, or use an off-the-shelf LLM for routing decisions? Once Maestro's RL training code is open-sourced, it can be directly used to fine-tune routing strategies.

**[KVServe: Service-Aware KV Cache Compression for Communication-Efficient Disaggregated LLM Serving](https://huggingface.co/papers/2605.13734)**
🔬 Breakthrough: It overturns the assumption that "KV Cache compression ratio should be fixed", proposing **dynamically adjusting the compression ratio based on current service load, bandwidth, and SLO**. In a PD-disaggregated architecture, compared to static compression (e.g., fixed 50% compression ratio), KVServe reduces P99 latency by about 40% under mixed loads, with the compression ratio dynamically fluctuating between 30% and 70%.
⚙️ Engineering Impact: For teams deploying PD-disaggregated inference, this means KV Cache compression is no longer a "set-and-forget" configuration item, but requires introducing a real-time monitoring + dynamic adjustment control plane. This week, evaluate: Has KV Cache transmission become a bottleneck in your current inference framework? If so, KVServe's "service-aware" approach is more economical than simply increasing bandwidth.

**[π-Bench: Evaluating Proactive Personal Assistant Agents in Long-Horizon Workflows](https://huggingface.co/papers/2605.14678)**
🔬 Breakthrough: It overturns the evaluation method of "Agent capability = single-turn instruction following", proposing **evaluating the Agent's ability to "proactively discover implicit needs" in long-horizon, multi-turn interactions**. In π-Bench's tests, the current strongest Agent (e.g., OpenClaw) has a proactive questioning rate of only about 35% in scenarios where "the user only says 'help me arrange a trip' but doesn't mention budget/preferences", compared to about 80% for human assistants.
⚙️ Engineering Impact: This means the current "passive execution" mode of Agents may be severely lacking in real-world scenarios. When building personal assistant Agents, teams need to design "intent mining" as an independent module, rather than relying on the user to clearly state all requirements at once. π-Bench's test set can serve as a benchmark for evaluating an Agent's "proactiveness".
